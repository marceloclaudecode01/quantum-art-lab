Why We Used Both Quantum Computers

The original plan was straightforward: generate all 100 quantum seeds on Origin Quantum's cloud. We chose Origin because their WK_C180 chip offers 180 superconducting qubits — one of the largest publicly accessible processors from a Chinese quantum hardware company — and the pyqpanda3 SDK was simple enough to integrate into our pipeline.

We minted NFTs #1 through #18 on Origin Quantum without issues. Then the chip entered a maintenance window, and queue times became unpredictable. We needed 82 more NFTs, and we needed them on a schedule.

Enter IBM Quantum. With its fleet of 100+ qubit processors, mature Qiskit ecosystem, and the SamplerV2 primitive, IBM was the natural scaling path. NFTs #19-100 were generated on ibm_fez (156 qubits) and ibm_torino (133 qubits).

The result is a collection that carries quantum provenance from two independent quantum computers on different continents. That wasn't the plan, but it turned out to be a feature.

Origin Quantum: Setup & Code

Origin Quantum's cloud is accessed through the pyqpanda3 SDK. Installation is a single pip command, but the documentation journey is not straightforward — you'll be reading Chinese docs and reverse-engineering examples.

pip install pyqpanda3

Here's the core code we used to generate quantum seeds on WK_C180:

import os, hashlib
from pyqpanda3 import QCloudService, QProg, H, CNOT, measure

# Connect to Origin Quantum Cloud
service = QCloudService(
    api_key=os.environ["QCLOUD_API_KEY"],
    url="http://pyqanda-admin.qpanda.cn"
)
backend = service.backend("WK_C180")

# Build max-entropy circuit: H + CNOT on 12 qubits
num_qubits = 12
prog = QProg()

for i in range(num_qubits):
    prog << H(i)                    # Hadamard: equal superposition
for i in range(0, num_qubits - 1, 2):
    prog << CNOT(i, i + 1)          # Entangle adjacent pairs

# IMPORTANT: pyqpanda3 requires explicit qubit AND cbit lists
prog << measure(list(range(num_qubits)), list(range(num_qubits)))

# Execute on real quantum hardware (8000 shots)
result = backend.run(prog, shots=8000)

# Convert measurement counts to deterministic seed
raw_bits = "".join(f"{k}:{v}" for k, v in sorted(result.items()))
seed = hashlib.sha256(raw_bits.encode()).hexdigest()
print(f"Quantum seed: {seed}")

Gotcha #1: The URL must be http://pyqanda-admin.qpanda.cn — not the public website URL (qcloud.originqc.com.cn). This isn't obvious anywhere in the docs. I found it by reading the SDK source code.

Gotcha #2: In pyqpanda3, the measure() function requires both qubit indices AND classical bit indices explicitly — measure([0,1], [0,1]). This is a breaking change from the older pyqpanda package and is barely documented.

IBM Quantum: Setup & Code

IBM's ecosystem is more polished. Sign up at quantum.ibm.com, grab your API token, install two packages, and you're running circuits on real hardware in minutes.

pip install qiskit qiskit-ibm-runtime

Our IBM quantum seed generation code:

import os, hashlib
from qiskit import QuantumCircuit
from qiskit_ibm_runtime import QiskitRuntimeService, SamplerV2
from qiskit.transpiler.preset_passmanagers import generate_preset_pass_manager

# Connect to IBM Quantum
service = QiskitRuntimeService(
    channel="ibm_quantum_platform",
    token=os.environ["IBM_QUANTUM_TOKEN"]
)
backend = service.least_busy(min_num_qubits=12, operational=True)

# Build the same H + CNOT max-entropy circuit
qc = QuantumCircuit(12, 12)
for i in range(12):
    qc.h(i)
for i in range(0, 11, 2):
    qc.cx(i, i + 1)
qc.measure(range(12), range(12))

# Transpile for target hardware topology
pm = generate_preset_pass_manager(backend=backend, optimization_level=1)
transpiled = pm.run(qc)

# Execute with SamplerV2 primitive (4096 shots)
sampler = SamplerV2(mode=backend)
job = sampler.run([transpiled], shots=4096)
result = job.result()

# CRITICAL: SamplerV2 uses 'c' register, NOT 'meas'
counts = result[0].data.c.get_counts()

raw_bits = "".join(f"{k}:{v}" for k, v in sorted(counts.items()))
seed = hashlib.sha256(raw_bits.encode()).hexdigest()
print(f"Quantum seed: {seed}")

Gotcha #1: The channel name recently changed from ibm_quantum to ibm_quantum_platform. Old tutorials will trip you up.

Gotcha #2: SamplerV2 results use the classical register name .c, not .meas. This changed from SamplerV1 and breaks code copied from older guides.

Key difference: IBM requires client-side transpilation to map your abstract circuit onto the physical qubit topology. Origin handles this server-side. IBM's approach gives you more control and visibility; Origin's is simpler but opaque.

Results: Randomness Quality Compared

Both platforms produce genuine quantum randomness. Our circuit puts 12 qubits into maximal superposition (Hadamard gates) and entangles them in pairs (CNOT gates), then measures. This produces a probability distribution across 4,096 possible bitstrings (2^12).

We hash the measurement counts with SHA-256 to produce a deterministic 64-character hex seed for each NFT. The quality of randomness from both chips is statistically indistinguishable — as it should be, since both are executing the same quantum physics.

Both Origin Quantum and IBM Quantum produce measurement distributions that pass chi-squared uniformity tests. The differences between platforms are in developer experience and tooling, not in the quality of quantum randomness.

Origin's 8,000 shots per job produce smoother distributions than IBM's 4,096, but both provide more than enough entropy for cryptographic-quality seed generation. Each NFT's seed has 256 bits of entropy — far beyond what any pseudorandom generator could match in provability.

Developer Experience Breakdown

Where IBM Wins

  • Documentation: Comprehensive English docs, interactive tutorials, a free textbook, and YouTube content
  • Tooling: Qiskit ecosystem is massive — circuit visualization, transpilation analysis, error mitigation libraries
  • Community: Stack Overflow answers, Discord server, active GitHub issues with maintainer responses
  • Stability: The API is mature, well-versioned, and rarely breaks unexpectedly
  • Backend selection: service.least_busy() finds an available machine in one line
  • Debugging: Descriptive error messages, real-time queue dashboard, job status tracking

Where Origin Wins

  • Qubit count: 180 qubits on WK_C180 vs IBM's 156 on ibm_fez
  • Shot count: 8,000 shots per job on free tier vs IBM's 4,096
  • Execution speed: Jobs complete in 5-10 seconds when the queue is clear
  • Simplicity: No client-side transpilation needed — submit and get results
  • Unique provenance: Hardware that most Western developers will never access

Pain Points

Origin's biggest challenge is documentation. The SDK works, but it's sparsely documented in English. I had to read Python source code to discover that measure() requires explicit classical bit indices. Error messages sometimes arrive in Chinese. The API URL isn't documented anywhere obvious. If you're comfortable navigating unfamiliar territory, it's fine. If you need hand-holding, it's frustrating.

IBM's biggest friction is API churn. The SamplerV2 primitive changed how results are accessed. The channel name changed. Many online tutorials are outdated. Queue times can be frustrating for batch jobs — we occasionally waited 5 minutes between NFT generations. But the documentation always catches up, and the community provides answers quickly.
